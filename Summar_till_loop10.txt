## 1️⃣ Project Overview

The **AI GitHub Agent** is a Python system that understands GitHub repositories, answers natural-language questions about them with cited sources, and (once Loops 10-13 complete) will propose code fixes as draft pull requests always reviewed by humans.

### Three engineering paradigms combined:

* **RAG (Retrieval-Augmented Generation)** — retrieve relevant code chunks and feed them to an LLM as context, so answers are grounded in your actual code.
* **MCP (Model Context Protocol)** — a JSON-RPC standard from Anthropic that lets any compatible client (Claude Desktop, custom UIs) call your agent's tools.
* **Agent reasoning** — a Planner that turns plain-English goals into structured, multi-step execution plans (and Loop 10 will actually run them).

### Two architectural pipelines:

* **Offline indexing pipeline** — clone a repo, parse it, chunk it, embed it, store it. Runs once per repo (or on-demand refresh).
* **Online runtime agent pipeline** — user request → planner → (Loop 10 executor) → retrieval + tools → LLM → (Loop 12 validator) → (Loop 13 draft PR).

### 🛡️ Core safety principle

The agent **never** merges code. Every proposed change goes through validation and lands as a **draft pull request** that a human reviews before merging. This is deliberately "autonomous but safe."

## 2️⃣ Architecture

### 📥 Offline Pipeline (Indexing)

```
GitHub Repo
    ↓
Clone (GitPython)
    ↓
AST Parser (Python ast module)
    ↓
Function/Class/Method Chunks (verbatim source + line numbers)
    ↓
Metadata Extraction (SHA-256 hash + imports + calls + module path)
    ↓
Embedding Model (sentence-transformers/all-MiniLM-L6-v2, 384-dim)
    ↓
Vector Store (pure-Python NumPy, cosine similarity, JSON persistence)
```

### 🌐 Online Pipeline (Runtime Agent)

```
User
    ↓
Frontend / API (FastAPI)
    ↓
Planner (Loop 9: creates execution plan)
    ↓
[PLANNED Loop 10: Execution Engine]
    ↓
(Hybrid Retrieval + MCP GitHub Tools + Session Memory)
    ↓
Context Assembly & Compression
    ↓
LLM
    ↓
[PLANNED Loops 12-13: Validation Manager, Confidence, Human Approval, Draft PR]
```

### Key design decisions

* **Two pipelines are decoupled** — indexing runs offline; runtime just queries the already-indexed vector store. This is how real production RAG systems work.
* **Pluggable backends everywhere** — LLMClient, GitHubAPI, Cloner, Planner, EmbeddingService are all Protocols. Real vs fake for tests. Easy to swap providers later.
* **Frozen Pydantic models** — CodeChunk, Plan, Answer, etc. are immutable. Prevents corruption downstream and makes debugging predictable.
* **Human-in-the-loop by default** — the plan template for 'fix a bug' has explicit HUMAN\_APPROVAL and DRAFT\_PR steps at the end.

## 4️⃣ Complete Concept Map

Ten technologies you should recognize in your code and explain in your own words. **Read once before any meeting.**

### 1. Python Virtual Environment (.venv)

* 🎯 **What:** An isolated folder where Python packages live so different projects don't conflict.
* 🤔 **Why:** So FastAPI, pytest, and everything else exist only inside this project — not polluting your system Python.
* 📍 **Where:** The `.venv/` folder at the project root, activated with `.\.venv\Scripts\Activate.ps1`.
* 💬 **Say:** *"I use a virtual environment so each project has its own isolated set of packages — nothing leaks between projects."*
* 🆘 **Deeper:** *"I know how to create, activate, and install into it. The deeper Python import resolution I'm still learning."*

### 2. FastAPI

* 🎯 **What:** A modern Python framework for building web APIs — handles HTTP requests and returns responses.
* 🤔 **Why:** Industry standard for Python AI/ML APIs, auto-generates docs at /docs, uses type hints natively.
* 📍 **Where:** `backend/main.py` — defines all 7 endpoints and the create\_app factory.
* 💬 **Say:** *"FastAPI turns my Python code into a runnable web server. I have 7 endpoints using the application-factory pattern."*
* 🆘 **Deeper:** *"I understand decorators, dependency injection, response models. The async event-loop internals I'm still learning."*

### 3. Pydantic + pydantic-settings

* 🎯 **What:** A library that validates Python data using type hints — wrong types crash early with clear errors.
* 🤔 **Why:** Type-safe config (Settings) and type-safe models (CodeChunk, Plan, Answer). Bugs caught early instead of in production.
* 📍 **Where:** `backend/config.py` (Settings), `backend/indexing/models.py`, `backend/agent/models.py`, `backend/rag/models.py`.
* 💬 **Say:** *"Pydantic enforces types at runtime. I used it for config, CodeChunk, Plan, Answer, and more. It catches bugs early."*
* 🆘 **Deeper:** *"I understand Field, validators, frozen models, SecretStr. Custom serializers and complex generics I'm still learning."*

### 4. SecretStr

* 🎯 **What:** A Pydantic type that hides secret values from logs, error messages, and print statements.
* 🤔 **Why:** So API keys (OpenAI, GitHub, Gemini) never accidentally show up in stack traces or logs.
* 📍 **Where:** `backend/config.py` — wraps `openai_api_key`, `gemini_api_key`, `github_token`.
* 💬 **Say:** *"I wrapped all API keys in SecretStr so they don't leak through logs. Production-grade habit from day one."*
* 🆘 **Deeper:** *"That's pretty much the whole story — it hides the value unless you explicitly call .get\_secret\_value()."*

### 5. Application Factory Pattern

* 🎯 **What:** Instead of creating the FastAPI app at the top of a file, you wrap it inside a `create_app()` function.
* 🤔 **Why:** Makes the app trivially testable (each test gets a fresh instance) and avoids global-state issues.
* 📍 **Where:** `backend/main.py` — the `create_app()` function with `app = create_app()` at the bottom.
* 💬 **Say:** *"I used the application-factory pattern. It's a FastAPI best practice — clean testing, no globals."*
* 🆘 **Deeper:** *"I know why it's better than a top-level app for testing. Multi-environment config injection I'm still learning."*

### 6. pytest + Fixtures

* 🎯 **What:** The standard Python testing framework. `tmp_path` fixture gives each test a fresh temp directory automatically.
* 🤔 **Why:** To prove every loop's code actually works — no regressions creep in as the project grows.
* 📍 **Where:** All files in `tests/` — currently **202 passing tests**.
* 💬 **Say:** *"I use pytest to validate every module. Right now 202 passing tests across 9 loops."*
* 🆘 **Deeper:** *"I know fixtures (tmp\_path, monkeypatch), assertions, TestClient. Mocking, parametrize, async fixtures I'm still learning."*

### 7. Python AST (Abstract Syntax Tree)

* 🎯 **What:** Standard-library module that turns Python source into a tree you can walk and analyze.
* 🤔 **Why:** To extract functions, classes, methods as clean chunks ready for embedding and retrieval.
* 📍 **Where:** `backend/indexing/ast_parser.py` and `backend/indexing/metadata.py`.

### 8. sentence-transformers

* 🎯 **What:** A library that turns text into fixed-length vector embeddings using pre-trained transformer models.
* 🤔 **Why:** To make code chunks semantically searchable — similar meaning, close vectors.
* 📍 **Where:** `backend/indexing/embeddings.py` — all-MiniLM-L6-v2, 384-dim.

### 9. RAG (Retrieval-Augmented Generation)

## 5️⃣ Big Picture Flows (in words)

### 💬 Flow A: When someone asks a question (Loop 5 world)

```
USER
  → uvicorn serves the request
  → FastAPI route POST /qa
  → RAGPipeline.ask()
  → VectorStore.query() returns top-k chunks
  → assemble_context() packs them under a budget with citation numbers
  → LLMClient.complete() generates the answer
  → JSON Answer with question, answer text, sources, model name, used_context_chars
```

### 📥 Flow B: When you index a new repo (Loop 8 world)

```
POST /index with a GitHub URL
  → GitCloner.clone() (skip if cached)
  → parse_directory() walks every .py file into CodeChunks
  → extract_metadata_for_chunks() adds hash + imports + calls + module path
  → VectorStore.add_chunks() embeds + stores with metadata
  → IndexResult with files_scanned, chunks_indexed, was_cached
```

### 🧠 Flow C: When you ask the agent to plan work (Loop 9 world)

```
POST /plan with goal='Fix login bug'
  → route picks RuleBasedPlanner or LLMPlanner
  → planner returns a Plan with typed steps:
    (retrieve → github_read → analyze → generate → validate → human_approval → draft_pr)
  → Loop 10 will actually execute these
```

### 🤖 Flow D: When an MCP client calls a tool (Loops 6/7/8/9 world)

```
POST /mcp with JSON-RPC request
  → MCPServer.handle() dispatches by method:
    - initialize → returns server info
    - tools/list → returns 8 tool schemas
    - tools/call with name+arguments → runs the tool
  → Tool errors come back as isError=true content blocks (MCP convention)
  → Protocol errors come back as JSON-RPC errors
```

### Deeper technical Q\&A

**Q: Why did you pick FastAPI?**
A: Async-native, auto-generates OpenAPI docs, works beautifully with Pydantic for typed request/response. It's the de facto standard for Python ML/AI APIs. I could have used Flask, but FastAPI's built-in dependency injection and validation saved me a lot of glue code.

**Q: Why did you build your own vector store instead of using Chroma?**
A: I evaluated ChromaDB first but its Rust backend has an access violation crash on Python 3.14. Rather than downgrade Python, I built a small NumPy vector store with cosine similarity and JSON persistence. Same public API, so I can swap back to Chroma or move to Qdrant / LanceDB later without touching any call sites. That's a Protocol pattern — defining the interface separately from the implementation.

**Q: How does the planner work?**
A: Two implementations behind a Planner protocol. RuleBasedPlanner picks one of five templates by keyword matching — fast, deterministic, testable. LLMPlanner asks the model to return JSON in a strict shape, then parses it, with graceful fallback to rule-based if the JSON is malformed. The route lets the caller pick strategy per request.

**Q: What's a strategic tradeoff you made?**
A: I kept the vector store simple — in-memory with JSON persistence. That's fine for now: swappable interface, works in tests, no external dependencies. Would upgrade before scaling to thousands of repos. Choosing 'boring' over 'clever' when possible is a hallmark of production engineering.

**Q: What's still missing? Be honest.**
A: Execution engine (Loop 10) — my planner produces plans but nothing runs them yet. Fix generation (Loop 11), validation pipeline (Loop 12), and draft PR flow (Loop 13). I have architecture for all of them. Also missing: real hybrid retrieval (I do vector-only now), replanning under failure, and observability. Those are future loops.

**Q: What did you learn?**
A: Two big things. First — **tests are the deliverable**, not the code. Without 202 passing tests, I couldn't refactor confidently. Second — **Protocols make agents pluggable**. LLMClient, GitHubAPI, Cloner, Planner, EmbeddingService — all Protocols. Real vs fake per test. This is the pattern real agent systems use.
